1. Abstract

Urmeaza sa se prezinte procesul de dezvoltare al aplicatiei RealHome.

2. Stagiile de Dezvoltare

2.1 Inceputul

Dezvoltarea platformei a inceput cu realizarea unui prototip de interfata. Dupa ce UI-ul si UX-ul au fost definite prin mockup-uri realizate in Sketch (Software pentru design) a urmat construirea schemei tehnice.

Bazat pe cerintele proiectului, am ales stack-ul de lucru (HTML, CSS(SASS) si Javascript nativ) si ne-am concentrat asupra realizarii unui MVP, plecand de la cele stabilite anterior. Prototipul a fost concretizat in HTML, CSS si Javascript, folosind date generice pentru a fi afisate informatii

2.2 Implementarea Inferfetei Grafice

Acest pas a constat in transcrierea elementelor grafice in cod, respectand sematica HTML5

Stilizarea a fost facuta prin intermediul CSS-ului, post-procesat din format SASS. Am ales utilizarea acestui post-procesator de CSS din motive de scalabilitate si consistenta.

Dinamica paginii a fost realizata cu ajutorul Javascriptului, folosindu-ne de feature-urile puse la dispozitie de versiuna ES6 (fara o post-procesare cu Babel). Organizarea a fost facuta cu ajutorul modulelor si sistemului de import a claselor. Tot aici, am folosit fetch API pentru a pregati terenul requesturilor la API-ul RealHome (urmeaza).

2.3 Baza de date si server

La acest stagiu a fost proiectata baza de date si a fost populata cu date generice pentru a testa si a putea lega restul aplicatiei de aceste date.

Lipsa unui API public free pentru extragerea LIVE a unor liste de apartamente/case/birouri ne-a indemnat la scierea unui scraper minimalist pentru a prelua cateva date in scopul popularii bazei noastre de date pentru teste.

Baza de date este relationala, oferind o implementare MySQL, iar partea de script-uri a fost realizata in PHP 7 (la final, compatibilitatea a ramas si la nivel de PHP 5), cu ajutorul unui server cu configuratie clasica.

2.4. API-ul RealHome

La nivel de structura am ales imparirea clara a platforme intre Front si Back. Informatiile sunt deci furnizate de API-ul propriu, si organizate in pagina de sistemul independent "din fata". Astfel, in cazul in care ar fi fost nevoie vreodata de schimbarea sursei de informatii (e.g. schimbam din PHP in NodeJS), efortul sa fi fost minim.

2.5 Sistemele de inregistrare si logare a utilizatorilor

Token-ul este memorat sub forma unui JWT semnat. Acesta contine informatii precum Alfanumeric-ID-ul utilizatorului (daca este cunoscut) si valoarea token-ului de autentificare.

Memorarea se face prin intermediul unui Cookie, cu o perioada de viata prestabilita. Pentru a securiza procesul, pe langa criptarea clasica, token-ul are o perioada de valabilitate si de refresh. In momentul atingerii momentului de refresh, token-ul se va regenera si se va inlocui in cookie-ul utilizatorul. Astfel, marim securitatea platfromei si oferim si o navigare intuitiva, fara a fi nevoiti sa deconectam utilizatorul (daca acesta intra constant, token-ul se va regenera singur pana la infinit).

Sistemul de utilizator se bazeaza pe interpretarea a 2 entitati:

  1. Utilizatorul Inregistrat - detine cont pe platforma
  2. Utilizatorul Anonim - vizitator

Sistemul face tracking la utilizatori, indiferent de tipul lor pentru a oferi recomandari bazate pe experienta lor in platforma.

Acest tracking se face pe baza unui token unic ce contine date de identificare are utilizatorului curent.

2.6 Conectarea cu API-uri externe

Pentru a putea afisa si calcula informatii interesante din proximitatea locatiei (ex. nivel de securitate bazat pe numar de crime, nivel de "fun" bazat pe numar de cluburi, restaurante etc.) am recurs la folosirea unor API-uri externe publice

In aceasta etapa s-a lucrat pe integrarea aplicatiei cu API-uri externe, precum:

  • Google Maps API
  • Google Places API
  • NYC Open Data

2.7 Integrarea modulelor

Conform celor explicate la punctul 2.4. modulele au fost integrate relativ simplu. Totul fiind gandit modular, codul de integrare a fost doar plug-and-play.
